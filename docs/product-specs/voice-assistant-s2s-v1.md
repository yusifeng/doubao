# Voice Assistant S2S V1 Product Spec

## 1. 文档目标

定义 V1 的产品规格（what），用于指导后续技术方案设计（how）与执行计划拆解。

## 2. 业务目标

- 构建一个可稳定语音聊天的助手（语音输入 + 语音输出）。
- 对齐参考产品的核心体验：**一个会话页面，支持文字/语音两种模式切换，并通过抽屉切换会话**。
- 在不引入复杂能力的前提下，优先打通主链路并可稳定演示。

## 3. 范围与约束

### 3.1 In Scope（V1 要做）

- 会话主页面：展示消息流、文本输入、语音模式切换。
- 会话抽屉：展示历史会话，支持进入会话与新建会话。
- 设置页：支持回复链路模式切换、S2S 基础配置、LLM 配置、音色选择与连接测试。
- 语音模式：作为同一会话页面的沉浸形态，显示“听/想/说”状态，支持停止与退出。
- 实时语音对话主链路：用户说话 -> 识别 -> 助手回复文本 -> 助手播报语音。
- 样式 Tokenizer Design：建立统一设计令牌（颜色/间距/圆角/字体层级），页面样式仅消费令牌。

### 3.2 Out of Scope（V1 不做）

- 多角色市场与复杂人设编辑。
- 联网搜索、外部 RAG、插件系统。
- 多端同步与账号体系。
- 高级音色管理、唱歌模式等扩展能力。

## 4. 已确认的产品决策

基于本次需求对齐，V1 固定如下：

- 语音交互模式：**免长按**（非 push-to-talk）。
- 平台优先级：**Android 优先**。
- 助手形态：**通用助手（非强角色扮演）**。
- 页面模型：**单一会话页 + 会话抽屉 + 文字/语音双模式**。

## 5. 核心用户故事

- 作为用户，我希望打开 App 后可以直接进入最近一次会话，不必先经过会话列表页。
- 作为用户，我希望在同一条会话里自由切换打字和语音，而不会丢失上下文。
- 作为用户，我希望保留历史会话，方便回看之前内容并继续聊。
- 作为用户，我希望在网络波动或失败时看到可理解的提示并快速重试。

## 6. 页面与信息架构

### 6.1 会话主页面

- 主体：消息流、输入区、模式切换。
- 文字模式：文本输入框、发送按钮、语音入口按钮。
- 语音模式：助手头像或对话展示、状态提示、四键控制条。
- 进入语音模式后自动开始收听，不要求二次点击“开始通话”。
- 语音模式内部再分两种展示形态：
  - 头像展示：大头像 + 短状态文案
  - 对话展示：对话文本 + 短状态文案
- 右上角按钮用于在语音模式内部切换展示形态，不直接退出语音模式。
- 四键控制条语义：
  - 第一键：默认承担收音控制；当 assistant 正在播报时，优先作为“手动点击打断”
  - 第二、第三键：占位按钮
  - 第四键：退出语音模式并回到文字模式
- Android Dialog SDK 链路支持会话内静音：
  - 第一键在非播报阶段可切换静音/恢复收麦
  - 静音后不再接收外部声音输入，但会话保持在线
  - assistant 已开始播放的语音继续播放，不因静音被打断
- 手动点击打断的产品语义：
  - 立即停止当前 assistant 语音输出
  - 已经显示出的 assistant 文本保留，不再继续增长
  - 页面状态切回 `正在听...`
  - 当前 conversation 不结束
- 用户插话自动打断在 Android Dialog SDK 主链已支持：
  - assistant 播报中检测到用户再次开口，会自动触发打断
  - 保留已显示的 assistant 文本并继续当前会话上下文
  - 其他平台链路继续按后续增强节奏推进

### 6.2 会话抽屉

- 列表项字段：标题、摘要、更新时间。
- 交互：点击切换会话；支持新建会话；可做本地搜索过滤。

## 7. 状态模型（产品层）

统一状态机：

- `idle`：空闲可发起对话
- `listening`：正在听用户输入
- `thinking`：正在生成回复
- `speaking`：正在语音播报
- `error`：当前轮失败，用户可重试

状态切换要求：

- 界面状态与实际链路状态一致，避免“看起来在听，实际已断流”。
- 任一失败场景都要可恢复到 `idle`。
- 从语音模式切回文字模式时，不丢当前会话内容。

## 8. 语音事件流（产品语义）

对接字节 S2S Realtime / Android Dialog SDK 文本能力（参考文档）：
- [端到端实时语音大模型API接入文档](https://www.volcengine.com/docs/6561/1594356?lang=zh)

V1 默认事件流程（稳定优先）：

1. 本机语音识别开始（ASR）
2. 用户结束输入后生成最终识别文本
3. 将最终文本写入当前会话
4. 接收助手文本回复
5. 本机 TTS 播报
6. 回合结束并回到空闲态

兼容策略：

- 文字模式和语音模式共享同一 `conversationId`。
- 语音模式下产生的最终用户转写与助手回复必须回到同一消息流。

## 9. 非功能需求（V1）

- 首包反馈可感知（有明确“正在听/思考”提示）。
- 断线、超时、鉴权失败时有明确错误文案。
- 会话操作可回退：退出语音模式后不丢当前会话内容。
- 日志最小可观测：关键阶段有可排查记录（连接、识别、回复、播报、失败）。
- UI 令牌一致性：核心页面不允许散落硬编码视觉值（颜色/字号/间距）。

## 10. 验收标准

- Android 真机/模拟器可完成至少 5 轮连续语音对话。
- 用户无需长按即可开始一轮语音输入。
- 能看到并区分 `listening / thinking / speaking / error` 四类状态。
- 会话抽屉可见，点击后能回看消息流并切换会话。
- 失败后支持重试并可继续对话。
- 核心页面样式由统一 token 驱动，至少覆盖颜色与排版层级。

## 11. 待技术方案决策（下一阶段）

- 音频采集与播放 SDK 选型。
- 本地存储方案（会话/消息持久化）。
- 设置页结构与模型/音色配置入口。
- S2S 接入方式（直连或服务端中转）。
- Token 承载介质（NativeWind theme 扩展或独立 token 模块）。
